REPORTE DE PREPARACIÓN DE DATOS PARA CONSULTAS HETEROGÉNEAS CON LLM
================================================================================

Etapa documentada
-----------------
Esta etapa corresponde a la preparación de los recursos locales que serán usados en
la siguiente libreta de consultas heterogéneas con LLM. El objetivo no fue responder
todavía las 20 preguntas, sino dejar listos los tres insumos principales:

1. Una base SQLite para consultas tabulares.
2. Un índice TF-IDF por artículo para búsqueda textual y similitud temática.
3. Un índice TF-IDF por autor para comparar perfiles temáticos de autores.

La decisión metodológica fue mantener el modelo canónico limpio como fuente
principal, sin modificarlo, y crear recursos derivados para facilitar las consultas.
La estructura relacional se mantuvo simple: una sola tabla canónica en SQLite. Las
relaciones complejas autor-artículo-keyword-afiliación se consultarán después desde
los grafos NetworkX ya creados, y la similitud textual se resolverá con TF-IDF.


Archivos generados
------------------
La libreta fue configurada para generar únicamente tres archivos de salida:

1. UNAM_Completo_2024_2025.sqlite
   Base SQLite local con una sola tabla: articulos_autores.

2. tfidf_articulos.joblib
   Objeto serializado con el vectorizador TF-IDF, la matriz TF-IDF por artículo y
   los metadatos necesarios para recuperar los artículos encontrados.

3. tfidf_autores.joblib
   Objeto serializado con el mismo vectorizador TF-IDF, la matriz TF-IDF por autor
   y los metadatos necesarios para recuperar autores y sus temas agregados.

---------------------
Se creó una base SQLite con una sola tabla física:

Esta tabla conserva la estructura artículo-autor-afiliación de la base curada. Cada
fila representa una relación entre un artículo y un autor UNAM, junto con sus
metadatos bibliográficos.

Para mejorar consultas frecuentes se crearon índices internos de SQLite sobre:

Autor_norm

Estos índices no cambian los datos, sólo facilitan búsquedas, agrupaciones y filtros
que se usarán en la libreta de consultas.

La decisión de usar una sola tabla se tomó porque la base ya está limpia y en formato
canónico. Crear muchas tablas relacionales habría duplicado relaciones que ya están
representadas de mejor manera en los grafos NetworkX.

Preparación del índice TF-IDF
-----------------------------
Se creó una representación textual-vectorial local usando TF-IDF. Esta capa no
reemplaza a la base tabular ni a los grafos. Funciona como apoyo para las preguntas
que requieren búsqueda textual o similitud temática.

Campos usados:

Titulo
Keywords
Abstract

En vez de crear un índice separado para cada campo, se creó una representación
combinada ponderada. La ponderación usada fue:

Titulo x3
Keywords x4
Abstract x1

Justificación:

- Titulo identifica de forma compacta el tema del artículo.
- Keywords son el campo más importante para detectar similitud temática, por eso
  reciben el mayor peso.
- Abstract aporta contexto amplio, pero si se usa con demasiado peso puede dominar
  la búsqueda y diluir el efecto de las keywords.

Esta combinación permite que la siguiente libreta pueda responder preguntas sobre:

- artículos relacionados con una keyword;
- autores con temas similares;
- posibles candidatos de colaboración por cercanía temática;
- publicaciones parecidas o variantes;
- similitud entre autores a partir de los textos de sus artículos.


Configuración técnica de TF-IDF
-------------------------------
El vectorizador TF-IDF se configuró con:

Máximo de términos: 30,000
N-gramas: unigramas y bigramas
min_df: 1
max_df: 0.95
Stopwords: lista básica en español e inglés
Normalización: la propia de TfidfVectorizer

Se usó el mismo vectorizador para artículos y autores. Esto es importante porque
ambas matrices quedan en el mismo espacio vectorial, lo que permitirá comparar:

consulta contra artículos;
artículo contra artículo;
autor contra autor;
consulta contra perfil temático de autor.

Decisiones metodológicas relevantes
-----------------------------------
1. Se mantuvo una sola tabla SQLite.
   La base original ya está curada y normalizada como artículo-autor, por lo que no
   era necesario dividirla en varias tablas. Esto simplifica la explicación y reduce
   errores en la generación de SQL.

2. No se alteró la base canónica.
   Los archivos creados son derivados para consulta. No modifican la base limpia.

3. Se creó TF-IDF combinado, no uno por campo.
   Para responder preguntas temáticas, conviene que título, keywords y abstract
   participen juntos, pero con pesos distintos.

4. Se creó TF-IDF por artículo y por autor.
   El índice por artículo permite recuperar publicaciones; el índice por autor permite
   comparar perfiles temáticos de investigadores.

5. Se evitó crear archivos innecesarios.
   La libreta genera sólo los tres recursos necesarios para la siguiente etapa.
